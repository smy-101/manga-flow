## Context

当前 Reader 页面是单页翻页模式，每次只渲染一张 `<img>`，通过 `convertFileSrc()` 加载。翻页时旧图片消失、新图片开始加载，存在白屏闪烁。页面导航只能逐页翻，无法快速跳转。只有一种阅读模式。

关键约束：
- 图片通过 Tauri `asset://` 协议加载，不走 HTTP
- Reader 是全屏 fixed 布局，无 Layout 包裹
- 状态管理用 Zustand，阅读进度存 SQLite
- 项目不写页面级测试

## Goals / Non-Goals

**Goals:**
- 消除翻页时的白屏等待
- 提供快速跳转到任意页面的能力
- 新增连续滚动阅读模式作为单页模式之外的选项

**Non-Goals:**
- 双页展开、RTL、手势翻页、缩放裁边等高级功能
- 阅读模式设置持久化（本次用默认值）
- 后端 Rust 改动（纯前端变更）

## Decisions

### 决策 1：预加载策略 — 隐藏 `<img>` 预加载

**选择**：在 DOM 中创建隐藏的 `<link rel="preload">` 或隐藏 `<img>` 元素，提前加载当前页前后各 2 页（共 5 页窗口）。

**备选方案**：
- A) Service Worker 缓存：Tauri WebView 对 Service Worker 支持有限，不适合
- B) `Image()` 对象预加载：可行但需要额外管理加载状态，不如直接用 DOM 元素直观

**理由**：浏览器对 `<img>` 的缓存机制天然有效。预加载的图片一旦加载完成，后续 `<img>` 引用同一 URL 时直接命中缓存，无白屏。实现简单，无需额外依赖。

### 决策 2：页码滑条 — 使用原生 `<input type="range">`

**选择**：底部固定定位的 range input，宽度占满屏幕，样式自定义。

**备选方案**：
- A) 第三方滑条组件（如 rc-slider）：增加依赖，功能过剩
- B) 自定义拖拽实现：开发成本高，无必要

**理由**：原生 range input 功能完全满足需求，通过 CSS 自定义外观即可匹配暗色主题，零依赖。

### 决策 3：连续滚动模式 — 虚拟化长列表

**选择**：使用虚拟滚动（仅渲染可视区域 ± buffer 的页面），所有页面纵向堆叠在滚动容器中。

**备选方案**：
- A) 全量渲染所有 `<img>`：100+ 页的漫画会导致 DOM 过多、内存占用高
- B) 分块加载：复杂度高，体验不如虚拟滚动流畅

**理由**：虚拟滚动兼顾性能和体验。但考虑到漫画页通常是高分辨率大图，即使虚拟化，同时加载的图片也不宜过多。采用简单的"视口 + 上下各 2 页 buffer"策略，不引入第三方虚拟滚动库。

### 决策 4：模式切换 UI — 顶部工具栏按钮

**选择**：在现有顶部工具栏中添加阅读模式切换按钮（图标按钮），点击在单页/连续滚动之间切换。

**理由**：最小 UI 改动，复用现有工具栏。模式切换是低频操作，不需要快捷键。

### 决策 5：Reader 组件拆分

**选择**：将 Reader.tsx 拆分为子组件：
- `SinglePageViewer`：当前的单页翻页视图
- `ContinuousScrollViewer`：连续滚动视图
- `PageSlider`：页码滑条控件
- `ReaderToolbar`：顶部工具栏（已有，提取为组件）

Reader.tsx 作为容器组件，根据模式渲染不同 Viewer。

**理由**：Reader.tsx 已经不小了，加上新功能会更臃肿。拆分后每个组件职责单一，可独立测试。

## Risks / Trade-offs

- **[内存占用] → 预加载窗口 5 张图**：漫画页图片较大（2-5MB/张），5 张约 10-25MB。桌面应用内存宽裕，可接受。后续可调整为 3 张（前后各 1 页）。
- **[连续滚动性能] → 简易虚拟化**：不自建复杂虚拟滚动引擎，而是用 IntersectionObserver 监测可见性，只渲染可见 ± buffer 的图片。足够满足漫画阅读场景。
- **[模式切换时的进度同步] → 统一 currentIndex**：两种模式共享同一个 `currentIndex`，切换模式时自动滚动/跳转到对应位置。
