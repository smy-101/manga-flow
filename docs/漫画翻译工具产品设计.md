# 漫画翻译工具产品设计（含管理与阅读）

## 一、产品定位
一个本地优先的漫画翻译与阅读一体化工具：支持导入漫画、自动翻译、排版修复、并提供良好的书库管理与阅读体验。

---

## 二、核心用户流程
1. 导入漫画（文件夹 / 压缩包）
2. 自动建库（识别章节与页序）
3. 选择章节或单页进行翻译
4. 生成翻译版本（覆盖或副本）
5. 在阅读器中浏览（原文 / 译文切换）
6. 管理书库（标签、进度、收藏）

---

## 三、系统模块划分

### 1. 书库管理（新增核心模块）
功能：
- 扫描与导入：支持 .zip/.cbz/.rar（可先只做 zip/cbz）
- 自动结构解析：书名 / 章节 / 页序
- 元数据：作者、标签、语言、封面
- 阅读进度记录（按书/按章节/页码）
- 收藏 / 最近阅读

数据存储：
- SQLite（Tauri 本地）
- 表设计建议：
  - books(id, title, cover, path, language, tags)
  - chapters(id, book_id, title, order)
  - pages(id, chapter_id, index, file_path)
  - progress(book_id, chapter_id, page_index, updated_at)


### 2. 阅读器（新增核心模块）
功能：
- 基础阅读：单页 / 双页 / 连续滚动
- 缩放与拖拽
- 快捷翻页（键盘/手势）
- 显示模式：
  - 原图
  - 翻译图
  - 对照（左右或覆盖切换）
- 文本气泡高亮（可选）

技术实现：
- 前端 Canvas / WebGL 渲染
- 虚拟列表（大批量页优化）


### 3. 翻译流水线（核心）
- OCR（云优先 → 本地化）
- 文本翻译（LLM / DeepL）
- 区域擦除（基础填充 → inpainting）
- 文本重绘（自动换行 + 字体适配）

输出：
- 新图片（译文层）
- 或“图层数据”（进阶：可切换显示）


### 4. 渲染与排版引擎
- 文本布局（气泡内自适应）
- 字体策略（漫画风字体）
- 方向支持（横排/竖排）


### 5. 任务系统
- 批量翻译章节
- 队列 + 进度条
- 失败重试


### 6. 设置与扩展
- API Key 管理
- OCR/翻译服务切换
- 字体与样式配置

---

## 四、技术架构

### 前端（Tauri + React）
- UI：书库、阅读器、翻译面板
- 状态：Zustand / Redux
- 渲染：Canvas

### 后端（Rust）
- 文件系统（解压/扫描）
- 数据库（SQLite）
- 图像处理（image / opencv）

### AI能力（初期走云）
- OCR：Google / PaddleOCR 服务化
- 翻译：OpenAI / DeepL

---

## 五、版本规划（结合阅读功能）

### V1（可用原型）
- 导入本地漫画（文件夹）
- 基础书库列表
- 简单阅读器（翻页）
- 单页翻译（云 OCR + 翻译）

### V2（体验完善）
- 章节结构解析
- 阅读进度保存
- 批量翻译
- 原/译切换

### V3（进阶）
- 本地 OCR
- 图像修复（inpainting）
- 更真实排版

### V4（产品级）
- 智能气泡检测
- 多语言支持
- 插件系统

---

## 六、关键难点（更新）

1. 漫画 OCR 准确率
2. 排版还原（影响观感）
3. 阅读性能（大图+多页）
4. 书库结构解析（杂乱文件命名）

---

## 七、实现建议（很关键）

开发顺序建议：
1. 先做书库 + 阅读器（能看漫画）
2. 再接入“单页翻译”
3. 再做批量与排版优化

原因：
- 阅读体验是长期使用的基础
- 翻译只是增强功能

---

## 八、一个现实建议

如果你一开始就把“翻译 + 排版 + 阅读器 + 管理”一起做，很容易崩。

更稳的路径是：
- 先做一个“好用的本地漫画阅读器”
- 再把“AI翻译”作为外挂能力接进去

这样产品更容易成型，也更容易持续迭代。
